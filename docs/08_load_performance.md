# Этап 8. Оценка нагрузки / лимитов

**Файл:** `docs/08_load_performance.md`

## 1. Сценарии сетевой нагрузки

### S1. Открытие сайта и загрузка нового источника
- 1 HTTP‑запрос к RSS‑источнику.
- 1 чтение/запись в SQLite.

### S2. Переключение между 3 источниками подряд
- До 3 HTTP‑запросов, если кэш пуст или истёк.
- Если кэш актуален, сетевых запросов может не быть.

### S3. Нажатие “Обновить” несколько раз подряд
- Каждый запрос может инициировать новый HTTP‑вызов.
- Нужно ограничить повторные обновления, чтобы не спамить RSS‑источник.

## 2. Оценка частоты запросов
Худший учебный сценарий:
- 1 пользователь,
- 3 источника,
- пользователь 5 раз подряд нажимает “Обновить” для каждого источника за 1 минуту.

Расчёт:
- `3 источника x 5 обновлений = 15 запросов в минуту`.

Для учебного проекта это умеренная нагрузка, но для открытых RSS‑источников лучше уменьшить её за счёт защиты от повторов.

## 3. Меры защиты
- TTL кэша: 15 минут.
- Кнопка “Обновить” блокируется на **2 секунды** после нажатия.
- Пока источник находится в состоянии Loading, повторный запрос к нему не запускается.
- Для HTTP‑запросов используется таймаут **5 секунд**.
- При неуспешном обновлении используется последний сохранённый снимок ленты.

## 4. Ожидаемая производительность
- Рендер страницы из кэша: до **2 секунд**.
- Рендер после живого запроса RSS: до **5 секунд** при нормальной сети.
- Фильтрация по уже загруженным новостям выполняется без сетевого запроса.
