# Этап 7. Конфигурация, хранение данных, ограничения и политика ошибок

**Файл:** `docs/07_config_storage_constraints.md`

## 1. Конфиг-параметры и значения по умолчанию
- `default_source_id = "weird-news"`
- `items_per_page = 20`
- `cache_ttl_minutes = 15`
- `network_timeout_seconds = 5`
- `use_cache_on_error = true`
- `max_sources = 5`

## 2. Где храним локальные файлы / данные
Для учебного проекта допустимы два варианта:

### Вариант A. SQLite в каталоге проекта
- `data/app.db` — база SQLite
- таблицы: `news_items`, `feed_snapshots`, `settings`

### Вариант B. SQLite в пользовательском каталоге
- Windows: `%APPDATA%/KrinzhLenta/app.db`
- Linux/macOS: `~/.local/share/krinzhlenta/app.db`

В первой версии можно использовать вариант A, чтобы проект было проще проверять преподавателю.

## 3. Политика кэша
- Кэшируем результат загрузки RSS по каждому источнику.
- Ключ кэша: `source_id`.
- TTL: **15 минут**.
- Если пользователь нажал “Обновить”, `force_refresh=true`, и кэш игнорируется.
- Если сеть недоступна, но есть неистёкший или последний успешный кэш, показываем его с предупреждением.

## 4. Ограничения / лимиты
- Внешние RSS‑источники могут ограничивать частоту запросов.
- Один и тот же источник не опрашивается автоматически чаще, чем 1 раз в 15 минут без явного действия пользователя.
- Максимум отображаемых новостей на странице — 100.
- Максимальное число источников в учебной версии — 5.
- Поддерживаются только HTTP/HTTPS RSS‑ленты.

## 5. Политика ошибок

### Что показываем пользователю
- Короткое понятное сообщение без технического мусора.
- При наличии кэша показываем предупреждение, например: **“Показана последняя сохранённая версия ленты.”**

### Что логируем
- Время ошибки.
- Тип ошибки (`NetworkError`, `InvalidFeedFormatError`, `StorageError`).
- URL RSS‑источника.
- HTTP‑статус или текст исключения.
- Краткий stack trace только для режима разработки.
